:::info Архитектурный обзор
Система предназначена для приёма событий из ДБО, анализа подозрительной активности, создания инцидентов и ведения аудита.
:::

## Назначение

Архитектура системы обеспечивает:

1. приём и обработку событий из каналов ДБО;
2. хранение транзакционных и аналитических данных;
3. автоматическое выявление подозрительных операций;
4. формирование и обработку инцидентов;
5. работу аналитика службы ИБ;
6. формирование отчётности и аудита.


## Основные участники

| Участник | Назначение |
| --- | --- |
| Система ДБО |  источник событий |
| Модуль передачи данных | Принимает события от ДБО |
| Модуль приёма данных | Валидирует и нормализует события |
| База данных | Хранит транзакционные данные и инциденты |
| Модуль правил | Выполняет проверку событий и расчёт риска |
| Система мониторинга | Координирует обработку событий |
| Модуль управления инцидентами | Создаёт и ведёт инциденты |
| Интерфейс аналитика | Отображает инциденты и решения |
| Журнал аудита | Хранит историю действий |
| Аналитик / старший аналитик | рассматривает инциденты и принимает решения |
| Служба ИБ | настраивает правила и контролирует результаты |
| ИТ-подразделение | поддерживает инфраструктуру, интеграции и доступность |
| Регуляторы | получают отчётность в установленном порядке |

## Поток обработки

1. ДБО передаёт событие в модуль передачи данных.
2. Модуль приёма данных проверяет и сохраняет событие.
3. Модуль правил анализирует событие.
4. При превышении порога создаётся инцидент.
5. Аналитик рассматривает инцидент и фиксирует решение.

## Хранилища

1. **Реляционная БД** — пользователи, транзакции, сессии, правила, инциденты, решения.
2. **Key-Value хранилище** — сессионные данные.
3. **Аналитическое хранилище** — аудит и отчётность.

## Асинхронное взаимодействие

Асинхронно передаются события из ДБО в систему мониторинга и сообщения о создании инцидентов. Это позволяет обрабатывать поток операций без ожидания ответа от всех компонентов.

## Надёжность

1. повторная доставка сообщений;
2. обработка дубликатов;
3. DLQ для некорректных событий;
4. логирование ошибок и алерты;
5. идемпотентная обработка входящих сообщений.

## Безопасность

1. передача данных по защищённым каналам;
2. шифрование данных при хранении;
3. разграничение доступа по ролям;
4. аудит всех действий.

## Границы MVP

В MVP реализуются:

1. приём событий;
2. валидация и хранение;
3. правила детекции;
4. создание инцидентов;
5. ручной обзор;
6. базовая отчётность.

## Sequence Diagram автоматического обнаружения подозрительной операции

```plantuml
@startuml
title Автоматическое обнаружение подозрительной операции

participant "Мобильное приложение\n(ДБО)" as MobileApp
participant "Модуль передачи\nданных" as TransferModule
participant "Модуль приёма\nданных" as IngestModule
participant "База данных" as DB
participant "Модуль правил" as RuleModule
participant "Система мониторинга" as Monitor
participant "Модуль управления\nинцидентами" as IncidentMgr

MobileApp -> TransferModule : ОтправитьСобытие(транзакция)
activate TransferModule

TransferModule -> IngestModule : ПередатьСобытие
activate IngestModule

note right of IngestModule
  Валидация формата и обязательных полей:
  id_события, id_пользователя, сумма
end note

alt Данные невалидны
    IngestModule -> DB : ЗаписатьБД
    DB --> IngestModule : подтверждение_записи
    IngestModule --> TransferModule : ack: ошибка_формата
    TransferModule --> MobileApp : СообщитьОшибка("неверные данные")
    deactivate IngestModule
else Данные валидны
    IngestModule -> DB : СохранитьСобытие(событие)
    DB --> IngestModule : подтверждение_сохранения
    IngestModule -> Monitor : СобытиеСохранено(id_события)
    deactivate IngestModule

    Monitor -> RuleModule : ПоручитьОценку(id_события)
    activate RuleModule

    RuleModule -> DB : ПолучитьКонтекст(id_пользователя, окно)
    DB --> RuleModule : связанныеСобытия
    RuleModule -> RuleModule : РассчитатьОценкуРиска()
    RuleModule --> Monitor : ОценкаГотова(id_события, оценка)

    alt оценка < ПорогСРЕДНИЙ
        RuleModule --> DB : ОбновитьСтатус(id_события, "норма")
        deactivate RuleModule
    else оценка >= ПорогСРЕДНИЙ and < ПорогВЫСОКИЙ
        RuleModule -> IncidentMgr : СоздатьИнцидент(id_события, оценка, краткое_описание)
        activate IncidentMgr
        IncidentMgr --> RuleModule : id_инцидента
        IncidentMgr -> DB : СохранитьИнцидент(инцидент)
        DB --> IncidentMgr : подтверждение_сохранения
        IncidentMgr -> Monitor : ИнцидентСоздан(id_инцидента, приоритет=СРЕДНИЙ)
        deactivate RuleModule
        deactivate IncidentMgr
    else оценка >= ПорогВЫСОКИЙ
        RuleModule -> IncidentMgr : СоздатьИнцидент(id_события, оценка, "высокий приоритет")
        activate IncidentMgr
        IncidentMgr -> DB : СохранитьИнцидент(инцидент)
        DB --> IncidentMgr : подтверждение_сохранения
        IncidentMgr -> Monitor : ИнцидентСоздан(id_инцидента, приоритет=ВЫСОКИЙ)
        deactivate RuleModule
        deactivate IncidentMgr
    end
end
@enduml
