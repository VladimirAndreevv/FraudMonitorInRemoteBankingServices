# C4-модель архитектуры

## Назначение

C4-модель используется для последовательного описания архитектуры системы на нескольких уровнях детализации. Такой подход позволяет показать не только внешнее окружение системы, но и её внутреннюю структуру, взаимодействие компонентов и распределение ответственности между сервисами.

Для системы мониторинга подозрительной активности C4-модель помогает понять:

1. какие участники взаимодействуют с системой;
2. какие внешние сервисы и хранилища используются;
3. как происходит обработка транзакционных событий;
5. из каких контейнеров и компонентов состоит решение;
5. каким образом организована обработка инцидентов и аналитики.


## Уровень 1 — System Context Diagram

Диаграмма контекста показывает систему как единое целое и описывает её взаимодействие с внешними пользователями и сервисами.

В контексте решения участвуют:

1. клиент банка, выполняющий операции через систему ДБО;
2. аналитик службы информационной безопасности;
3. сотрудники службы ИБ;
4. система дистанционного банковского обслуживания;
5. регуляторы;
6. аналитическое хранилище данных.

Система мониторинга получает события от ДБО, выполняет анализ операций, рассчитывает уровень риска, формирует инциденты и предоставляет интерфейс для аналитиков. Дополнительно система передаёт агрегированные данные в аналитическое хранилище и формирует отчётность для регуляторов.


```plantuml

@startuml
!includeurl https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml

title C4 Context — Система мониторинга подозрительной активности в ДБО
LAYOUT_TOP_DOWN()

Person(client, "Клиент банка", "Совершает операции в ДБО")
Person(analyst, "Аналитик ИБ", "Рассматривает инциденты")
Person(sec, "Служба ИБ", "Настраивает правила детекции")
Person(regulator, "Регулятор", "Получает отчётность")

System(dbo, "Система ДБО", "Источник транзакционных событий")
System(monitoring, "Система мониторинга подозрительной активности", "Выявляет подозрительные операции, создаёт инциденты и поддерживает расследование")

System_Ext(corebank, "Ядро банка", "Хранение и обработка операций")
System_Ext(reporting, "Система отчётности", "Формирование регуляторных отчётов")

Rel(client, dbo, "Инициирует операции")
Rel(dbo, monitoring, "Передаёт события о транзакциях")
Rel(monitoring, analyst, "Показывает инциденты")
Rel(sec, monitoring, "Настраивает правила")
Rel(monitoring, corebank, "Запрашивает данные по операциям")
Rel(monitoring, reporting, "Формирует отчёты")
Rel(regulator, reporting, "Получает отчётность")

@enduml

```


## Уровень 2 — Container Diagram

Диаграмма контейнеров показывает основные части системы и их взаимодействие между собой.

Архитектура включает следующие контейнеры:

| Контейнер | Назначение |
| --- | --- |
| API Gateway | Точка входа для REST-запросов и интеграций |
| Kafka | Асинхронная передача событий |
| Модуль приёма данных | Валидация и нормализация событий |
| Модуль правил | Анализ транзакций и расчёт риска |
| Модуль управления инцидентами | Создание и сопровождение инцидентов |
| Web UI аналитика | Пользовательский интерфейс |
| PostgreSQL | Хранение транзакционных и инцидентных данных |
| Redis | Сессионные данные и быстрый доступ |
| ClickHouse / DWH | Аналитика и отчётность |

Поток обработки выглядит следующим образом:

1. Система ДБО публикует событие в Kafka.
2. Модуль приёма данных получает событие, выполняет проверку и сохраняет данные.
3. Модуль правил анализирует операцию и рассчитывает уровень риска.
4. При необходимости создаётся инцидент.
5. Аналитик работает с инцидентом через Web UI.
6. Аналитические данные передаются в DWH для отчётности и мониторинга.

```plantuml

@startuml
!includeurl https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

title C4 Container — Система мониторинга подозрительной активности в ДБО
LAYOUT_TOP_DOWN()

Person(client, "Клиент банка", "Совершает операции в ДБО")
Person(analyst, "Аналитик ИБ", "Рассматривает инциденты")
Person(sec, "Служба ИБ", "Настраивает правила детекции")

System_Boundary(sys, "Система мониторинга подозрительной активности") {
    Container(api, "Модуль приёма данных", "API / Ingestion Service", "Принимает и валидирует события из ДБО")
    Container(rules, "Модуль правил", "Rule Engine", "Выполняет анализ и расчёт риска")
    Container(incident, "Модуль управления инцидентами", "Incident Service", "Создаёт, хранит и закрывает инциденты")
    Container(ui, "Интерфейс аналитика", "Web UI", "Отображает инциденты и позволяет принимать решения")
    ContainerDb(db, "База данных", "PostgreSQL", "Хранит пользователей, транзакции, инциденты, решения")
    Container(audit, "Журнал аудита", "DWH / ClickHouse", "Хранит историю действий и агрегированные данные")
    Container(queue, "Очередь событий", "Kafka", "Асинхронная передача событий")
}

System_Ext(dbo, "Система ДБО", "Источник событий")
System_Ext(reporting, "Система отчётности", "Получает агрегированные данные")

Rel(client, dbo, "Инициирует операции")
Rel(dbo, api, "Передаёт события")
Rel(api, queue, "Публикует события")
Rel(queue, rules, "Передаёт события на анализ")
Rel(rules, db, "Читает и обновляет данные")
Rel(rules, incident, "Создаёт инциденты")
Rel(incident, db, "Сохраняет инциденты")
Rel(analyst, ui, "Работает с инцидентами")
Rel(ui, incident, "Получает данные инцидентов")
Rel(sec, ui, "Настраивает правила")
Rel(ui, rules, "Управляет правилами")
Rel(incident, audit, "Пишет события аудита")
Rel(rules, audit, "Пишет результаты анализа")
Rel(audit, reporting, "Передаёт агрегаты")

@enduml

```

## Уровень 3 — Component Diagram

Диаграмма компонентов показывает внутреннюю структуру модуля анализа подозрительных операций.

Внутри модуля правил используются следующие компоненты:

| Компонент | Назначение |
| --- | --- |
| Event Receiver | Получение событий из Kafka |
| Event Validator | Проверка структуры и обязательных полей |
| Normalizer | Нормализация данных |
| Rule Detector | Выполнение правил детекции |
| Risk Engine | Расчёт уровня риска |
| Incident Creator | Формирование инцидентов |
| Audit Logger | Запись действий и событий |

Обработка события происходит поэтапно:

1. Событие поступает из Kafka.
2. Выполняется проверка обязательных полей и структуры.
3. Данные приводятся к внутреннему формату.
4. Модуль правил выполняет анализ операции.
5. Рассчитывается уровень риска.
6. При превышении порога создаётся инцидент.
7. Все действия фиксируются в журнале аудита.

```plantuml
@startuml C4_Component
!includeurl https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Component.puml

LAYOUT_TOP_DOWN()
title Component Diagram — Модуль анализа подозрительных операций

System_Boundary(system, "Система мониторинга подозрительной активности") {
    Container_Boundary(rules, "Модуль правил") {

        Component(receiver, "Event Receiver", "Kafka Consumer", "Получение событий из Kafka")
        Component(validator, "Event Validator", "Validation Component", "Проверка структуры и обязательных полей")
        Component(normalizer, "Normalizer", "Transformation Component", "Нормализация входных данных")
        Component(detector, "Rule Detector", "Rule Engine", "Проверка правил детекции")
        Component(risk, "Risk Engine", "Business Logic", "Расчёт уровня риска")
        Component(incidentCreator, "Incident Creator", "Service", "Создание инцидента")
        Component(audit, "Audit Logger", "Logging Component", "Запись действий и событий")
    }
}

ContainerDb(pg, "PostgreSQL", "Database", "Хранение инцидентов и результатов анализа")
ContainerDb(redis, "Redis", "Key-Value Store", "Контекст сессии и быстрый доступ")
Container(kafka, "Kafka", "Message Broker", "Передача событий")

Rel(kafka, receiver, "Передаёт событие")
Rel(receiver, validator, "Передаёт данные")
Rel(validator, normalizer, "Корректные события")
Rel(normalizer, detector, "Нормализованные данные")
Rel(detector, risk, "Запуск анализа")
Rel(risk, redis, "Получение контекста")
Rel(risk, incidentCreator, "Создание инцидента")
Rel(incidentCreator, pg, "Сохранение инцидента")
Rel(audit, pg, "Запись аудита")

@enduml
```


## Назначение уровней C4

| Уровень | Назначение |
| --- | --- |
| System Context | Показывает систему и её окружение |
| Container | Описывает основные сервисы и хранилища |
| Component | Показывает внутреннюю структуру модулей |
