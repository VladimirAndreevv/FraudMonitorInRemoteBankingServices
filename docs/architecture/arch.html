Архитектура системы

Архитектурный обзор

Система предназначена для приёма событий из ДБО, анализа подозрительной активности, создания инцидентов и ведения аудита.

Назначение​

Архитектура системы обеспечивает:

  1. приём и обработку событий из каналов ДБО;
  2. хранение транзакционных и аналитических данных;
  3. автоматическое выявление подозрительных операций;
  4. формирование и обработку инцидентов;
  5. работу аналитика службы ИБ;
  6. формирование отчётности и аудита.

Основные компоненты​

КомпонентНазначение
Модуль передачи данныхПринимает события от ДБО
Модуль приёма данныхВалидирует и нормализует события
База данныхХранит транзакционные данные и инциденты
Модуль правилВыполняет проверку событий и расчёт риска
Система мониторингаКоординирует обработку событий
Модуль управления инцидентамиСоздаёт и ведёт инциденты
Интерфейс аналитикаОтображает инциденты и решения
Журнал аудитаХранит историю действий

Поток обработки​

  1. ДБО передаёт событие в модуль передачи данных.
  2. Модуль приёма данных проверяет и сохраняет событие.
  3. Модуль правил анализирует событие.
  4. При превышении порога создаётся инцидент.
  5. Аналитик рассматривает инцидент и фиксирует решение.

Хранилища​

  1. Реляционная БД — пользователи, транзакции, сессии, правила, инциденты, решения.
  2. Key-Value хранилище — сессионные данные.
  3. Аналитическое хранилище — аудит и отчётность.

Асинхронное взаимодействие​

Асинхронно передаются события из ДБО в систему мониторинга и сообщения о создании инцидентов. Это позволяет обрабатывать поток операций без ожидания ответа от всех компонентов.

Надёжность​

  1. повторная доставка сообщений;
  2. обработка дубликатов;
  3. DLQ для некорректных событий;
  4. логирование ошибок и алерты;
  5. идемпотентная обработка входящих сообщений.

Безопасность​

  1. передача данных по защищённым каналам;
  2. шифрование данных при хранении;
  3. разграничение доступа по ролям;
  4. аудит всех действий.

Границы MVP​

В MVP реализуются:

  1. приём событий;
  2. валидация и хранение;
  3. правила детекции;
  4. создание инцидентов;
  5. ручной обзор;
  6. базовая отчётность.
